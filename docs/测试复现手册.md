## 1. 测试基础设施(所有阶段共用的"假外部世界")

离线测试之所以能零成本又确定性,靠三件自建的测试替身(`tests/` 下)+ 注入式假时钟:

| 替身 | 文件 | 作用 |
|---|---|---|
| **MockLLM** | `tests/mock_llm.py` | 满足 `LLM` 协议,按**预设脚本**依次返回回复(str 或带 tool_calls 的 ChatResponse)。记录 `seen_messages`/`seen_systems`/`seen_tools` 供断言"上下文/system/工具是否正确下发"。脚本耗尽再调用会 AssertionError(暴露脚本写少了)。 |
| **MockEmbedder** | `tests/mock_embedder.py` | 满足 `Embedder` 协议,映射模式**指定谁和谁相似** + 哈希兜底 → 测试完全掌控向量相似度,不连 OpenAI。 |
| **InMemoryVectorStore** | `agent_framework/memory/vector_store.py` | 纯 Python 余弦检索,零依赖;既是测试替身,也是"存储可替换"的活证明。 |
| **假时钟 `now_fn`** | 各测试注入 | 三因子时近性、限流窗口、整任务超时都用注入的固定/可推进时钟,断言可稳定复现。 |

**核心思想**:凡是"会变的外部依赖"(LLM/Embedding/向量库/时钟)都在接口后,测试注入假的,
业务代码一行不改就能离线跑——这既是可测试性,也是可拓展性的同一套设计。

## 3. 阶段二 · 最小 Agent 循环(ReAct)

**测试文件**:`tests/test_agent.py`(14 个函数)· 基础设施 `tests/mock_llm.py`

```bash
python -m pytest tests/test_agent.py -q          # 14 passed
```

**覆盖的关键路径**(全部用 MockLLM 脚本化驱动,确定性):

| 覆盖点 | 验证什么 |
|---|---|
| 直接回答 | 第一步就 `final_answer` → 一步返回 |
| 单工具 | 查订单 → 作答 |
| **多工具链**(核心) | `query_order` → `query_logistics` → 作答(两步链) |
| 解析失败恢复 | 坏 JSON → 喂回"解析失败" → 下一步正确 → 成功恢复 |
| 未知工具恢复 | 调不存在的工具 → 喂回可用工具列表 → 恢复 |
| 工具异常恢复 | 工具 `run` 抛异常 → 喂回错误 → 恢复 |
| 撞 max_steps | 步数上限 → 触发**强制作答**,`stopped_reason="max_steps"` |
| parse_step 单测 | 代码围栏容忍、二选一校验(action/final_answer 恰有其一) |

**为什么这样测**:ReAct 的价值在"多步 + 自我纠正",所以三类错误恢复各一条、步数守卫一条——
证明**任何坏输入都不会让程序崩,循环有刹车**。

**手动复现**(可选,交互式):
```bash
python -m examples.react_cli           # /trace 看每步 Thought/Action/Observation
```

## 4. 阶段三 · Tool Use 系统

**测试文件**(4 个,共约 57 个函数):
- `tests/test_tools.py`(26)— BaseTool 执行管线 + strict mode + Registry + @tool 装饰器
- `tests/test_tool_calling.py`(10)— 原生 Function Calling 循环 + 双协议适配
- `tests/test_jd_tools.py`(13)— 8 个 JD 业务工具的正常/异常/状态机
- `tests/test_common_tools.py`(8)— calculator(AST 安全)/ current_time / http_request(白名单)

```bash
python -m pytest tests/test_tools.py tests/test_tool_calling.py \
                 tests/test_jd_tools.py tests/test_common_tools.py -q     # ~57 passed
```

**覆盖的关键点**:

| 类别 | 验证什么 |
|---|---|
| 校验-正常/strict/宽松 | 合法参数 ok=True;`"5"`→int 在 strict 下报错、宽松下强转;多余参数 strict 报错并列合法参数 |
| 执行-异常/超时 | `_run` 抛异常 → ok=False 不外泄;`timeout=0.1` + 慢 `_run` → ok=False 提示超时 |
| Schema 导出 | `to_schema()` 含 name/description/parameters;strict 时 `additionalProperties:false` |
| Registry | 注册/重名报错/replace 覆盖/get 未知抛错/invoke 未知返回 ok=False/批量导出 |
| @tool 装饰器 | Schema 从类型注解推断;**无注解/无 docstring 报错**;执行结果同手写类 |
| 兼容 | BaseTool 实例通过阶段二 Tool 协议检查,可直接插入 ReActAgent |
| 业务工具 | 订单查询/物流/商品;`cancel_order` 状态机(只对待发货生效,已发货拒绝,重复取消幂等);`apply_refund` 7 天无理由时效规则;查无≠错误(ok=True 提示文本) |
| 安全工具 | calculator **AST 白名单**(`__import__('os')` 被拒);http_request **域名白名单**(后缀伪装拦截) |
| 幂等(阶段六加) | 同 `request_id` 重复 invoke 返首次结果、副作用只发生一次(见 test_reliability) |

**为什么这样测**:每个工具"正常 ≥1 + 异常/查无 ≥1"是评审定的七条规范之一;strict mode 是
用户点名的硬要求,单列三种校验场景证明它真的拦住模型幻觉。

**手动复现**:

## 5. 阶段四 · Memory 与上下文管理

**测试文件**(5 个,共约 43 个函数)+ MockEmbedder:
- `tests/test_short_term.py`(9)— 滑窗轮原子性 + token 预算
- `tests/test_compressor.py`(6)— 递归摘要 + 失败降级
- `tests/test_long_term.py`(19)— 提炼/裁决/三因子/隔离(核心)
- `tests/test_memory_manager.py`(8)— 统一门面组装 + 跨会话
- `tests/test_chroma_store.py`(1,`importorskip`)— Chroma 落盘往返,**未装 chromadb 自动跳过**

```bash
python -m pytest tests/test_short_term.py tests/test_compressor.py \
                 tests/test_long_term.py tests/test_memory_manager.py \
                 tests/test_chroma_store.py -q       # ~43 passed(chroma 那条可能 skip)
```

**覆盖的关键点**(全离线,MockLLM + MockEmbedder + 假时钟):

| 机制 | 验证什么 |
|---|---|
| 轮原子性 | 窗口展开的消息里**不允许出现孤儿 tool 消息**;最新轮永不弹出;弹出轮由 add() 返回 |
| Token 预算 | 超预算按轮弹出(注意计的是 render_text 渲染文本,不是裸字符串) |
| 递归摘要 | 新摘要 = LLM(旧摘要+弹出轮);**LLM 抛异常 → 降级截断拼接不炸**(ExplodingLLM 用例) |
| 提炼 | 四规则;每轮 ≤3 条;JSON 非法 → 这轮不写 |
| **增删改语义** | ADD/UPDATE(地址北京→上海后库里只剩一条、id 不变内容换新)/DELETE(删旧不存新)/NOOP |
| **防误伤** | 裁决编造不在候选集的 target_id → 矫正为 ADD,旧记忆不动 |
| 三因子权重 | 构造"高相关但旧且不重要 vs 低相关但新且重要",验证权重翻转排序 |
| **隔离红线** | alice 的记忆 bob 检索必须 **0 命中** |
| 保鲜副作用 | 各场景用**独立 store**(第一次检索 touch 会污染第二个场景的时近性) |
| 降级全覆盖 | 提炼非 JSON / 裁决非 JSON / 压缩 LLM 异常,三处失败分支都是一等公民 |

**为什么这样测**:记忆最容易出隐蔽 bug(保鲜污染、隔离泄漏、裁决误删),所以隔离/防误伤/
降级各设红线用例;三因子权重用可解释的构造用例证明排序逻辑对。

### 5.1 检索准确率评测(交付物⑥,数据推翻了评审拍板)


**覆盖的关键点**:

| 模块 | 验证什么 |
|---|---|
| ReliableLLM | 可重试错误退避后成功(断言退避序列 [1,2,4]);不可重试错误直接上抛;重试耗尽抛最后异常 |
| FallbackLLM | 主失败切备;主健康不碰备;主备都挂上抛 |
| 幂等 | 同 request_id 只执行一次、store 只落一条;不同 id 分别执行;**失败结果不缓存**(改参后同 id 可再执行) |
| Tracer | 事件序完整;JSONL 落盘可回读;**listener 抛异常被吞**(旁路不拖垮业务) |
| on_event 钩子 | 事件序 llm_call/tool_call/tool_result/final_answer;**回调抛异常不破坏循环**;**无钩子行为不变** |
| metrics | 聚合数字正确(构造已知 trace 断言);空集合不除零 |
| 过滤/限流 | 注入模式标记;超长截断;密钥/手机号/身份证脱敏;正常文本零误伤;超频拒绝+窗口恢复;token 预算超限 |
| 边界标记 | BoundaryRegistry 只包成功结果;失败错误文本不包 |
| **审批闸门** | 高权限拦截入队、低权限透传;**approve 后真执行(幂等)**;重复审批抛错;规则配置驱动(medium 也能审批);未知工具折叠标准错误 |
| **人工队列** | 审批/升级两入口;**落盘重启可恢复**(request_id 不变);队列文件损坏从空开始不炸 |
| HITL 编排 | 三个升级入口各一条(重规划耗尽/质检二审不过/整任务超时);闸门×编排端到端(拦截→专员如实告知→放行真执行) |
| Judge | 四维打分 + clamp(9→5);**解析失败降级中性分**;A/B **交换顺序**消除位置偏见 |
| AgentService | direct 路由;快路径专员;**高权限被闸门拦**;出口脱敏;限流;注入标记 |
| API | /health;/chat 正常;**缺字段 422**;/chat/stream **SSE 三段事件**;/approvals 全流程;重复审批 400 |

**为什么这样测**:阶段六是"敢上线"的保障,所以每个可靠性机制(退避序列、幂等、降级)都精确
断言;HITL 是业务闭环核心,单独一个文件测三个升级入口 + 闸门端到端;API 用 TestClient 离线跑
