# 京东客服 Agent 框架 · 完整架构框架图

> 本文是整个框架的「地图」:一张请求全景图 + 九个子包的逐一拆解 + 外部数据接口清单 +
> 每个 Agent 的人设 + 控制系统设计。所有内容对应仓库真实代码(`agent_framework/`),
> 版本 0.6.0,六阶段全部落成。配套阅读:`docs/设计论文.md`(设计思路的纯文字详解)。

---

## 0. 一句话总览

**一条用户诉求进来,框架依次经过:入口安全 → 记忆装载 → 三级分诊 → 执行(专员 / 中心调度)
→ 出口安全 → 记忆落账,全程 Trace 记录;高权限动作与办不了的诉求统一转人工队列。**
核心哲学贯穿始终:**核心只认接口,加能力 = 加实现;全链路降级永不炸;凡是循环必有刹车;
结构化优于自由文本;安全的最后一道墙是最小权限。**

**怎么读这张图**:竖着看是**一次请求的处理链**(① 入口安全 → ⑥ 记忆落账);右侧
`◆ EXT-1~8` 是**八个外部数据接入点**,每个用 `◄──` 箭头指到它在链路里的接入位置。
所有接入点当前是 mock / 便宜实现,**生产替换只换实现、核心一行不动**(`↳ 生产:...` 标了换成什么)。

**外部数据到底在哪接入**(回答"哪里接外部数据"):
- **EXT-1 用户/前端**:请求从 `api/server.py` 进来,`{user_id, message}` —— user_id 是记忆隔离/限流/审批归属的键;
- **EXT-2 人工客服/工单系统**:高权限动作与办不了的诉求进 HandoffQueue,人工控制台放行后真正执行;
- **EXT-3 Embedding + EXT-4 向量库**:在 **② 记忆装载**处接入 —— 长期记忆的向量化(OpenAI)与存储(Chroma);
- **EXT-5 京东业务数据**:在 **④ 执行**的工具最内层接入 —— 现在是 `JDMockStore` 内存假数据,生产换成京东订单/商品/物流真实 API,11 个工具零改动;
- **EXT-6 外部 HTTP**:`http_request` 工具的域名白名单出站;
- **EXT-7 LLM API**:所有 LLM 调用经可靠性外壳落到 Claude/OpenAI,换厂商只改 `.env`;
- **EXT-8 可观测平台**:Tracer 事件流,当前落 JSONL,生产可对接 Langfuse(加一个 listener)。

**两条要点**:
1. **装饰链是架构精髓**:真实工具外面套三层包装(权限闸门 → 边界标记 → 工具),每层都实现
   Registry 的鸭子协议,专员和编排层完全不知道 registry 被包过——**加安全 = 加包装层,核心零改动**;
2. **外部依赖全在接口后**:EXT-3~7 分别是 Embedder / VectorStore / JDMockStore / http / LLM
   五个可替换接口,这正是"可拓展性硬要求"的兑现——换业务换实现而不动核心。

## 2. 可靠性外壳(所有 LLM 调用的底座)

无论哪条路径,所有 LLM 调用都经过同一层可靠性包装(`core/llm_reliable.py` + `create_llm`):

```
    上层(Agent / 编排 / 裁判)只见 LLM 接口
                    │
                    ▼
   ┌────────────────────────────────────┐
   │ FallbackLLM(可选,配了 fallback 才有)│  主 provider 挂 → 切备用
   └────────────────┬───────────────────┘
                    ▼
   ┌────────────────────────────────────┐
   │ ReliableLLM(装饰器)                 │  指数退避 1s→2s→4s + ±20% 抖动
   │  只重试可重试错误(超时/限流/5xx)     │  参数非法/鉴权失败直接上抛
   │  max_retries=3                       │
   └────────────────┬───────────────────┘
                    ▼
   ┌────────────────────────────────────┐
   │ ClaudeLLM / OpenAILLM(裸 provider)  │  SDK 只在这里 import(延迟)
   │  client 带 timeout(超时四层之②)     │
   └────────────────┬───────────────────┘
                    ▼
              外部 LLM API
```

## 3. 九个子包逐一拆解

### 3.1 `core/` — 运行时内核(阶段一、二 + 阶段六增强)

| 文件 | 职责 | 关键设计 |
|---|---|---|
| `llm.py` | LLM 抽象 + 工厂 | `LLM` Protocol(厂商无关)· `Message`/`ChatResponse`/`Usage`/`ToolCall` · `create_llm()` 装配可靠性包装 |
| `llm_claude.py` / `llm_openai.py` | 两家 provider 实现 | SDK 延迟导入;厂商协议差异(system 位置、token 字段、tool_use vs tool_calls)全封装在此 |
| `llm_reliable.py` | 可靠性包装 | `ReliableLLM` 退避重试 + `FallbackLLM` 降级(阶段六) |
| `config.py` | 配置中心 | `Settings`(pydantic-settings 从 .env 读)· 所有护栏参数可覆写 |
| `agent.py` | 两个 Agent 循环 | `ReActAgent`(文本 JSON,兜底)+ `ToolCallingAgent`(原生 Function Calling,优先)+ `on_event` 只读钩子 |

**两个 Agent 循环的分工**:
- `ToolCallingAgent`(主力):模型经 API `tools=` 参数返回结构化 `tool_calls`,无文本解析环节,支持一步并行多工具调用;
- `ReActAgent`(兜底):模型输出 JSON 文本 → `parse_step` 解析 → Pydantic 校验(强制 action/final_answer 二选一)→ 执行 → Observation 喂回;**任何只会吐文本的模型都能跑**。

两者共享 `AgentResult`(final_answer + steps 完整轨迹 + stopped_reason),都有 `max_steps` 刹车(默认 5,撞上限强制作答)。

### 3.2 `tools/` — 工具系统(阶段三 + 阶段六幂等)

```
BaseTool(ABC)                     ── invoke() 固定管线:校验(strict)→ 限时执行 → 标准化
 ├─ 声明:name/description/args_schema(Pydantic)/strict/timeout/permission/request_id 幂等
 ├─ 子类只写 _run() 业务逻辑
 └─ 任何失败折叠为 ToolResult(ok=False),永不抛异常
FunctionTool(BaseTool)            ── @tool 装饰器:函数一行变工具,Schema 从类型注解推断
ToolRegistry                      ── 注册/发现/统一调用/批量导出 Schema/subset() 切子集
JDMockStore                       ── 共享数据层(订单↔运单↔商品↔退款一致);接真 API = 换这层
```

**11 个工具及权限分级**(权限决定阶段六是否触发审批):

| 工具 | 权限 | 归属专员 | 说明 |
|---|---|---|---|
| query_order | 低 | 订单 / 售后 | 查订单状态与物流单号 |
| query_logistics | 低 | 订单 | 查包裹位置与预计送达 |
| query_user_orders | 低 | 订单 | 没报单号时先列历史订单 |
| current_time | 低 | 订单 | 「今天/现在」类问题 |
| query_product | 低 | 导购 | 商品价格/库存/保修(分词匹配) |
| search_faq | 低 | 售后 / 导购 | 平台规则(退换货/发票/优惠券) |
| calculator | 低 | 导购 | AST 白名单求值,不用 eval |
| http_request | 中 | — | 域名白名单 + 限时 + 响应体截断 |
| **apply_refund** | **高** | 售后 | 退款(含 7 天无理由时效校验)→ 审批 |
| **cancel_order** | **高** | 售后 | 取消未发货订单(状态机)→ 审批 |
| **create_ticket** | **高** | 售后 | 建人工工单 → 审批 |

**关键设计**:三个高权限工具**全部集中在售后专员**——阶段六的审批闸门只需卡住他一个。

### 3.3 `memory/` — 记忆与上下文(阶段四)

```
MemoryManager(统一门面,三件套皆可缺省)
 ├─ ShortTermMemory   短期滑窗(以「轮 Turn」为原子单位,tool 消息不拆散;
 │                     最新轮永不弹出;Token 预算默认 2000)
 ├─ SummaryCompressor 递归摘要(新摘要 = LLM(旧摘要 + 弹出轮),O(1) 增量;
 │                     失败降级截断拼接,永不炸对话)→ 前情提要
 └─ LongTermMemory    长期记忆
      ├─ 写入:LLM 提炼事实(四规则)→ Mem0 式 ADD/UPDATE/DELETE/NOOP 裁决
      ├─ 检索:三因子打分(相关性 1.0 + 时近性 0.25 + 重要性 0.25,权重经评测调优)
      │        命中记录"保鲜"(刷新 last_accessed_at)
      ├─ Embedder Protocol   → OpenAIEmbedder(text-embedding-3-small)
      └─ VectorStore Protocol → InMemoryVectorStore(测试)/ ChromaVectorStore(持久化)
```

**上下文四段分区**(MemGPT 灵感,预算紧张按优先级从低到高牺牲):
`system(永不裁)> 前情摘要(≤300)> 检索记忆(top_k=3)> 近期窗口(2000 tokens)`。

**多用户隔离在存储层强制执行**:search/list/delete 全部 user_id where 过滤,user_id 框架注入、不进 prompt、模型不可见。

### 3.4 `planning/` — 任务规划(阶段五)

```
Planner                           ── plan():诉求 → 线性步骤计划(每步指派专员)
 │                                    replan():某步失败 → 只重排剩余步骤(编号续接)
 │                                    降级:JSON 非法 → 单步计划(退化成单 Agent)
Plan / PlanStep                   ── 线性列表(非 DAG,客服步骤强依赖)
PlanExecutor                      ── 顺序执行 + 失败触发重规划(全局≤max_replans)
 │                                    stop_when 刹车(整任务超时/预算耗尽)
 │                                    依赖 StepRunner 协议(不 import multi_agent)
ScratchPad                        ── 步骤间共享黑板(结论追加,保新丢旧截断)
```

**依赖方向纪律**:`multi_agent → planning`,单向。executor 通过 `StepRunner` 协议解耦,由 Supervisor 提供适配器。

### 3.5 `multi_agent/` — 多 Agent 协作(阶段五)

```
Specialist(纯数据 + 工厂)          ── name/title/description/工具子集/专属 prompt
 │                                    内核就是 ToolCallingAgent(core 零改动)
TaskAssignment / TaskOutcome      ── 通信消息;失败前缀「无法完成:」= 协议状态码
Router(模式一)                    ── 入口分诊三出口:direct / 专员名 / supervisor
 │                                    降级:解析失败/未知目标 → supervisor
Supervisor(模式二)               ── 中心调度:按计划派工(非 LLM 自由选人)
 │                                    汇总 → Critic 质检 → 三处兜底升级转人工
Critic                            ── 终稿质检(四项检查单,对照证据)
 │                                    不合格回炉≤1;解析失败降级放行
render_roster                     ── 花名册(Router/Planner 选人依据)
```

### 3.6 `observability/` — 可观测(阶段六)

```

TraceEvent  ── task_id/seq/ts/kind/payload;kind 枚举 task_start/route/plan/step/
```

### 3.7 `safety/` — 安全与 HITL(阶段六,业务闭环核心)

```
input_filter    ── 清洗 + 注入检测(标记不硬拦)+ HARDENING_CLAUSE 加固条款
                   + BoundaryRegistry 工具返回边界标记(防间接注入)
output_filter   ── 敏感信息脱敏(密钥/手机号/身份证正则)
rate_limiter    ── RateLimiter 滑动窗口限流 + TokenBudget 单任务 token 预算
approval        ── HITL 三件:
                   · ApprovalGate 权限闸门(Registry 装饰器,规则配置驱动)
                   · HandoffQueue 统一人工队列(审批/升级两入口,JSON 落盘)
                   · approve → 真正执行挂起动作(request_id 幂等,重放安全)
```

## 4. 三个业务专员的人设(Agent 角色设计)

所有专员共享**公共 prompt 底座** = 阶段三客服三规则(指代解析 / 不重复索要 / 工具失败先核参数)
+ 阶段五协作条款(前序结论直接用 / 超职责明示转交 / 失败前缀)+ 阶段六安全加固条款,
再叠加各自的**域块人设**:

```
┌─────────────────────────────────────────────────────────────────────────┐
│ order_agent · 订单物流专员                                                 │
│  工具:query_order · query_logistics · query_user_orders · current_time    │
│  人设:负责订单状态、物流进度、历史订单的一切查询;只查询、不做任何修改操作。   │
│  关键行为:问"到哪了/什么时候到"时,必须查完订单接着查物流,开门见山答         │
│           "包裹现在到哪个城市/哪一站 + 预计何时到",不罗列没问的商品/金额。    │
│  权限:全部低权限,不触发审批。                                             │
├─────────────────────────────────────────────────────────────────────────┤
│ aftersales_agent · 售后专员                                                │
│  工具:query_order · apply_refund · cancel_order · create_ticket · search_faq│
│  人设:负责退款、取消订单、创建人工工单。执行任何操作前必须先 query_order      │
│        核实状态与条件;不符合条件如实说明并评估建工单转人工,绝不强行操作。     │
│  关键:**三个高权限工具全在他手里** → 阶段六审批闸门只需卡他一个。            │
│  权限:退款/取消/工单均为高权限,触发人工审批。                              │
├─────────────────────────────────────────────────────────────────────────┤
│ product_agent · 商品导购专员                                               │
│  工具:query_product · search_faq · calculator                            │
│  人设:负责商品信息与推荐。推荐要给理由(参数/价格对比);不清楚的参数如实      │
│        说不知道,绝不编造。                                                 │
│  权限:全部低权限。                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**专员协作拓扑 = 星型**:专员之间**不直接通话**,一切经编排器(Router/Supervisor)中转;
`query_order` 在订单/售后专员间有意重叠(售后操作前自行查证,不依赖跨专员喊话)。

**编排层的"隐形角色"**(非业务专员,但也是 Agent/LLM 调用):
- **Router 分诊员**:一次调用,判 direct/专员/supervisor;
- **Planner 规划师**:把复杂诉求拆成线性步骤并指派专员;
- **Supervisor 主管**:按计划派工、汇总面向用户的方案;
- **Critic 质检员**:对照证据审终稿,四项检查单;
- **Judge 裁判**(离线):评测时给答复四维打分,= Critic 的离线复用;
- **direct 客服**:轻量无工具,直接回复告知类/寒暄输入。

## 5. 控制系统设计(三级分流 + 循环护栏 + 兜底升级)

### 5.1 三级分流(成本递增,便宜的路先走)

```
用户诉求 → Router 分诊
   ├─ direct        无需查询/操作数据(自报信息、寒暄)→ 1 次 LLM 调用直接回复
   ├─ 专员直派       单域问题,不需先查证再操作     → ~4 次调用(快路径,带外层历史)
   └─ supervisor    跨域/多动作/需先查证再操作      → ~11-15 次调用(中心调度全套)
```

### 5.2 循环护栏总表(凡是循环必有刹车,全部 Settings 可配)

| 层 | 上限 | 触发后行为 |
|---|---|---|
| 专员内环 | `supervisor_specialist_max_steps` / `agent_max_steps` = 5 | 强制作答,ok=False 上报 |
| 计划长度 | `planner_max_steps` = 6 | prompt 约束 + 超长硬截断 |
| 动态重规划 | `max_replans` = 1 | 再失败如实进结果,不兜圈 |
| Critic 回炉 | `critic_max_retries` = 1 | 二审仍不过 → 放行留痕 |
| LLM 重试 | `llm_max_retries` = 3 | 退避后仍失败 → 上抛/切 fallback |
| 整任务超时 | `task_deadline_seconds` = 300 | 转人工(第④层保险) |
| 单任务 token | `task_token_budget` = 50000 | 超限终止 → 转人工 |
| 用户限流 | `rate_limit_per_minute` = 20 | 拒绝并提示稍后再试 |

**最坏情况单轮 LLM 调用数有界**(约 43 次封顶),不存在无限循环路径。

### 5.3 人工介入队列(HITL)— 一个队列,两个入口

```
                    HandoffQueue(JSON 落盘 data/,重启不丢)
                    /                                    \
        入口 A:审批(approval)                入口 B:升级(escalation)
        ApprovalGate 拦下高权限工具调用        Supervisor 三处兜底:
        → 挂起动作(工具名+参数+幂等 ID)        · 重规划耗尽仍失败
        专员收到「已提交审批单 X」继续干活       · 质检二审仍不合格(放行+复核)
                    \                          · 整任务超时/超预算
                     \                                  /
                      人工控制台(CLI + /approvals API)
                       list / show / approve / reject / close
                            │ approve
                            ▼
                  真正执行挂起的工具调用(request_id 幂等,重放安全)
```

**这就是业务场景「低权限自动办、高权限转人工」的完整闭环**:
低权限工具专员直接调用返回结果;高权限工具被闸门拦下、挂起、等人工放行后执行;
Agent 彻底办不了的诉求带轨迹升级人工。审批规则(哪级 permission 要审批)由
`ApprovalPolicy` 配置驱动,不写死 if/else。

## 6. 外部数据接口清单(哪里接什么数据)

框架所有"会变的外部依赖"都藏在接口后,当前用 mock / 便宜实现,生产替换不动核心:

| 接口 / 抽象 | 当前实现 | 生产应接入 | 替换方式 |
|---|---|---|---|
| **LLM**(`LLM` Protocol) | ClaudeLLM / OpenAILLM(gpt-5.4-mini) | 任意 Chat LLM | 加一个实现 + 改 `.env` 的 `PROVIDER` |
| **业务数据**(`JDMockStore`) | 内存 mock(订单/物流/商品/FAQ/退款) | 京东真实订单/商品/物流 API | 换数据层,11 个工具零改动 |
| **Embedding**(`Embedder` Protocol) | OpenAIEmbedder(text-embedding-3-small) | 任意向量化服务 | 加一个实现 |
| **向量库**(`VectorStore` Protocol) | Chroma(持久化)/ InMemory(测试) | Milvus/Pinecone/pgvector 等 | 加一个实现 |
| **记忆落盘** | `data/memory/`(Chroma 文件) | 挂卷 / 托管向量库 | Docker volume |
| **人工队列**(`HandoffQueue`) | JSON 文件 `data/handoff_queue.json` | 数据库 / 工单系统 | 换存储实现,接口不变 |
| **限流 / token 预算** | 进程内(RateLimiter/TokenBudget) | Redis 等共享存储 | 换实现,接口不变 |
| **可观测**(`Tracer` listener) | JSONL 落盘 + 终端 viewer | Langfuse / Phoenix | 给 Tracer 加一个 listener |
| **HTTP 出站**(`http_request`) | 域名白名单 mock | 真实外部 API | 配白名单 |

**外部输入 / 输出契约**:
- **输入**:`{user_id, message, mode?}`(user_id 是记忆隔离 + 限流 + 审批归属的键,框架注入,不进 prompt);
- **输出**:`{answer, task_id, route, handoff_ids, redactions, rate_limited}`,或 SSE 事件流
  (`event: step` 中间步骤 → `event: answer` → `event: done`);
- **业务语义输出**:不是单个答案,而是**解决方案**(办成的给凭据,没办成的给原因+替代方案+工单号)。

## 7. 数据 / 控制流时序(以复合客诉为例)

```
用户:"耳机(订单11111)坏了,查还能不能退,能退帮我退,再推荐替代品"
  │
  ├─[入口安全] 限流通过 · 注入检测:干净
  ├─[记忆] load(user_id) → 无历史,system 附加段为空
  ├─[分诊] Router → "supervisor"(跨域多动作)
  ├─[规划] Planner → 计划:①查订单11111状态 ②申请退款 ③推荐替代品
  ├─[执行] PlanExecutor 逐步派工:
  │    step-1 → 订单专员:query_order(11111) → "已签收,超7天"  [ScratchPad 记下]
  │    step-2 → 售后专员:apply_refund(11111) → 7天时效规则拒绝 → "无法完成:超时效"
  │       └─ 失败!触发 replan → 新计划:①建工单转人工 ②推荐替代品
  │    step-4 → 售后专员:create_ticket → 被 ApprovalGate 拦截 → 挂起入队(审批单)
  │       └─ 专员收到"已提交审批单 T-xxx",视为已妥善移交(不算失败)
  │    step-5 → 导购专员:query_product("耳机") → 推荐小米蓝牙耳机 Pro
  ├─[汇总] Supervisor → 面向用户的方案(逐项交代:超时效不能直接退/已转人工/替代品)
  ├─[质检] Critic 对照证据审 → 一审漏推荐 → 回炉重写 → 二审通过
  ├─[出口安全] 脱敏:无敏感信息
  ├─[记忆落账] 提炼事实:"用户订单11111耳机坏了求退+推荐"、"已建工单T-xxx跟进"(承诺)
  └─ 返回解决方案 + Trace(全程 15 次 LLM 调用,有界)
```

## 8. 一图看懂"六阶段如何叠成一个系统"

```
阶段一 core/llm  ─────────────────────────────────► 可切换 provider(FallbackLLM 降级兑现)
阶段二 core/agent(ReAct 循环 + max_steps)─────────► 专员内核 ToolCallingAgent
阶段三 tools(BaseTool+Registry+11工具+permission)─► 专员工具子集 + 审批闸门读 permission
阶段四 memory(短期/摘要/长期三因子/隔离)──────────► AgentService 装载 user 记忆
阶段五 planning + multi_agent(Router/Supervisor)─► 控制系统(三级分流 + 计划驱动)
阶段六 observability/safety/api/evaluation ────────► 生产外壳 + HITL 闭环 + 服务化
                                                     ▲
                            贯穿:核心只认接口 · 全链路降级 · 凡循环必刹车 ·
                                  结构化优于自由文本 · 最小权限是最后一道墙
```

每一阶段的产出都成为下一阶段的零件,最终由 `AgentService` 门面组装成整栈——
**这不是六个独立作业,是一个层层叠起、一以贯之的系统。**
