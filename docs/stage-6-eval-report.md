## 3. 典型失败分析(评测的价值在于暴露问题)

- **r01(记忆)**:两轮「自报地址 → 问地址」,第二轮走了 direct 答「无法查看」——
  因为评测 harness 用的是**空 MemoryManager**(无长期记忆,见 §4),不是 Agent 逻辑
  缺陷。带记忆的 `multi_agent_cli` 手动冒烟里此场景是通过的。
- **c01/c02(多专员)**:办成了(取消/转人工/推荐),但把「1-3 个工作日」说成
  「1-7」、未逐字说「超 7 天无理由期」——**完整性**而非**准确性**问题,属汇总 prompt
  可优化项。
- **b05(边界)**:模型主动提「可以用优惠券再省一点」,越过了「不私自改价」的红线——
  真实的合规风险点,应写进售后/导购 prompt 的边界条款(阶段六后续或答辩提出)。
- **一处评测驱动的真实修复**:首轮 s03 因 `query_product` 整串子串匹配,"Anker 充电器"
  匹配不到 "Anker 快充充电器 65W"(中间隔"快充")。改为**分词匹配**后复跑满分——
  与阶段四「评测推翻评审权重」同一方法论:**数字暴露问题 → 定位 → 修复 → 复跑验证**。

## 4. 局限性(如实声明)

- **评测未接长期记忆**:harness 为隔离与零外部依赖用了空 MemoryManager,故 memory
  类用例只能验证短期行为;r01 的失败是配置所致,非能力所致。接入 `create_memory_manager`
  后应通过(手动冒烟已验证)。
- **裁判自偏**:裁判与被评同款模型(gpt-5.4-mini),可能偏爱同源文风;更严谨应换更强
  模型或双裁判(judge.py 已留 A/B 交换顺序防位置偏见的实现)。
- **小样本波动**:25 条不足以给稳定百分比,数字的正确用途是**改动前后回归对比**,
  不是绝对结论——延续阶段四 16 条评测集写过的同一教训。
- **单跑随机性**:未多跑取中位数;生产评估应固定或平均。

## 5. 结论

76% 的通过率、安全 4.84 的高分,配上「失分集中在措辞完整度而非硬错误」的失败画像,
支撑一个可量化的结论:**框架在业务场景下功能完整、行为安全,主要改进空间在汇总
措辞与边界 prompt**。更重要的是这条 pipeline 本身——它把「感觉挺好用」变成了
「25 例 76% 通过、四维可量化、改动可回归」的工程语言。
