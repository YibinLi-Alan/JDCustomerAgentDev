# 技术点精讲(简明版设计论文)

> 这是 `docs/设计论文.md`(完整版)的浓缩版。**按"是什么 → 优点 → 怎么实现"三步**讲清
> 框架里每一个关键技术点,大白话为主,专业名词当场加解释。想看每个阶段完整的思考过程,
> 读完整版;想快速搞懂"每个技术为什么这么设计",读这篇。

**三条贯穿全局的信条**(理解了它们,下面每个设计就都顺了):
1. **核心只认接口,加能力 = 加实现**——会变的东西(模型、工具、数据库)都藏在"接口"后面,
   换掉它只需写个新实现,不动主流程。
2. **全链路降级,永不炸对话**——任何一步失败(模型抽风、工具报错、网络断)都折叠成一条"数据"
   继续走,不把异常抛给用户。
3. **凡是循环必有刹车**——每个会重复的地方(Agent 步数、重试、重规划)都有次数上限,杜绝死循环。

## 一、大脑:LLM 接入层

### 1. LLM 接口抽象(Protocol)
- **是什么**:`Protocol`(接口协议,只规定"必须有哪些方法",不管具体怎么写)。我们定义一个 `LLM`
  接口,只声明 `chat()`/`stream()` 两个方法。Claude、OpenAI 各写一个类去满足它。
- **优点**:上层代码(Agent、编排)只认这个接口,**换模型/换厂商时主流程一行不改**;只用一家的
  用户不必安装另一家的 SDK(延迟导入)。
- **怎么实现**:`core/llm.py` 定义接口 + `create_llm()` 工厂;`llm_claude.py`/`llm_openai.py` 各自
  把厂商差异(system 参数位置、token 字段名)封装在内部;换厂商只改 `.env` 的 `PROVIDER` 一行。

### 2. 可切换 provider + 降级(FallbackLLM)
- **是什么**:主模型挂了自动切到备用模型。
- **优点**:**部分服务优于完全罢工**——OpenAI 抽风时切到 Claude,用户无感。
- **怎么实现**:`FallbackLLM` 包住主备两个 `LLM`,主调用失败就调备用。因为两者都是同一个接口,
  包装层不关心背后是谁。

## 二、循环:Agent 内核

### 3. ReAct 循环
- **是什么**:ReAct(Reasoning + Acting,推理与行动交替)。Agent 一步步"想(Thought)→ 调工具
  (Action)→ 看结果(Observation)→ 再想",直到能给最终答案。这是让模型从"只会聊天"变成
  "会自己做事"的核心。
- **优点**:模型能**自主决定**下一步该查什么、办什么,多步解决复杂问题;而不是一问一答。
- **怎么实现**:`core/agent.py` 的循环:每步调模型 → 拿到它要调的工具 → 执行 → 结果喂回 → 下一步。
  用 `max_steps`(默认 5)刹车,撞上限强制作答。

### 4. 结构化输出 + 错误恢复
- **是什么**:让模型每步输出规定格式(JSON),我们解析后执行;解析失败/工具报错时,把错误当成
  一条"观察结果"喂回去让模型自己纠正,而不是让程序崩。
- **优点**:**坏输出能自愈**——模型格式写错了,下一步看到报错就改对了;程序永不因为模型抽风而崩。
- **怎么实现**:Pydantic(数据校验库)强制"要调工具或要作答,二选一";解析/未知工具/工具异常
  三类错误都折叠成文本喂回,且都计入步数(所以坏输出也不会无限循环)。

### 5. 原生 Function Calling(与文本解析双模式)
- **是什么**:Function Calling(函数调用),模型 API 层直接返回"要调哪个工具、参数是什么"的
  结构化数据,不用我们从文本里解析。
- **优点**:**没有解析环节**(更稳)、支持一步**并行调多个工具**;文本解析版作为兜底,让任何
  只会吐文字的模型也能跑。
- **怎么实现**:`ToolCallingAgent`(原生,优先)+ `ReActAgent`(文本解析,兜底)两个循环并存,
  共享同一套结果结构。厂商协议差异(Claude 的 `tool_use` block、OpenAI 的 `tool_calls`)封装在
  各自 provider 文件里。

## 三、手脚:工具系统

### 6. BaseTool + 统一执行管线
- **是什么**:所有工具的抽象基类(ABC,抽象基类:规定子类必须实现哪些方法)。子类只写业务逻辑
  `_run()`,校验/超时/兜错全由基类的 `invoke()` 管线统一处理。
- **优点**:**加一个工具 = 写一个类**,不用重复写校验和错误处理;失败一律折叠成 `ToolResult`
  数据,循环永不崩。
- **怎么实现**:`invoke()` 固定三步——参数校验 → 限时执行 → 结果标准化,任何一步失败返回
  `ToolResult(ok=False)`。

### 7. strict mode(严格模式)
- **是什么**:工具参数校验时**禁止类型强转、拒绝多余参数**(比如模型把订单号写成数字、或多传了
  一个不存在的参数,直接报错)。
- **优点**:工具参数全来自模型生成的 JSON,严格模式让模型的**幻觉(编造)立刻暴露**并被喂回纠正,
  而不是被静默容错掩盖。
- **怎么实现**:Pydantic 的 `strict=True` 校验 + Schema 里标 `additionalProperties: false`(不许加
  未知字段);接 OpenAI 时映射到它的 `strict: true`(服务端保证输出严格合规)。

### 8. ToolRegistry(注册中心)+ @tool 装饰器
- **是什么**:工具的注册、发现、统一调用中枢;`@tool` 装饰器能把一个普通函数一行变成工具。
- **优点**:Agent 不用拿一堆散装工具,拿一个 Registry 即可;**加能力 = 注册一个实现,核心零改动**;
  `@tool` 让写小工具的成本极低。
- **怎么实现**:Registry 内部是 `name → 工具` 的字典;重名默认报错(装配 bug 炸在装配时);`@tool`
  用 `create_model` 从函数的类型标注自动推断参数 Schema,没写 docstring 直接报错。

### 9. 权限分级(permission)
- **是什么**:每个工具标一个权限级别(低/中/高)。查询类是低,退款/取消/建工单是高。
- **优点**:这是"分级执行"的技术基础——**声明与拦截分离**,阶段六的审批闸门只需读这个标记就能
  决定"直接执行还是转人工"。
- **怎么实现**:`BaseTool.permission` 元数据,阶段三只声明不拦截,阶段六 `ApprovalGate` 读取它。

## 四、记忆

### 10. 短期记忆:滑动窗口 + 轮原子性
- **是什么**:滑动窗口(只保留最近 N 轮对话,超了就把最老的挤出去),防上下文无限膨胀撑爆 token。
  以"轮"为最小单位(user + 工具往返 + assistant 算一轮),不从中间切断。
- **优点**:上下文可控(省钱、不超模型窗口);**不拆散工具消息**(否则会留下"孤儿"消息让 Claude
  报 400 错)。
- **怎么实现**:`ShortTermMemory` 按 token 预算(默认 2000)裁剪整轮;最新轮永不弹出;弹出的轮交给
  压缩器,信息不凭空丢。token 用启发式估算(中文 1 字≈1 token),故意不引 tiktoken(那个只对
  OpenAI 准)。

### 11. 上下文压缩:递归摘要
- **是什么**:把挤出去的旧对话摘要成一段"前情提要"。递归(recursive)= 新摘要 = 模型(旧摘要 +
  刚弹出的几轮),每次只摘增量。
- **优点**:成本 O(1)(每次只处理新增的,不重摘全部历史);近期原文 + 早期摘要的混合,既省 token
  又不丢重要信息。
- **怎么实现**:`SummaryCompressor`;模型调用失败时降级为"旧摘要 + 弹出轮文本"的截断拼接,**永不
  炸对话**。

### 12. 长期记忆:embedding + 三因子检索
- **是什么**:embedding(嵌入,把文字转成一串数字向量,语义相近的向量也相近)。检索时不只看
  相关性,用三个因子加权打分:相关性 + 时近性(越近越高)+ 重要性。出自 Generative Agents 论文
  (斯坦福 AI 小镇)。
- **优点**:纯按相关性检索会漏掉"虽然语义一般但很新/很重要"的记忆;三因子更贴合真实需求(比如
  用户最近说的地址应该优先)。
- **怎么实现**:`score = 相关性 + 0.25·时近性 + 0.25·重要性`(权重是**评测数据调出来的**——论文
  的等权在客服场景只有 38% 命中,轻量档 94%);命中的记忆刷新访问时间("保鲜")。存储走
  `VectorStore` 接口,Chroma(持久化,重启还在)/ 内存版可替换。

### 13. 记忆写入:Mem0 式增删改
- **是什么**:不存对话原文,而是让模型**提炼事实**(记身份/诉求/承诺,不记工具能查的易变状态);
  新事实入库前先召回相似旧记忆,让模型裁决 ADD(新增)/ UPDATE(覆盖)/ DELETE(删除)/ NOOP
  (不动)。出自 Mem0 开源记忆层。
- **优点**:"我搬家了"能**真正覆盖旧地址**而不是留两条打架;记忆是活的、会被修正,不无限堆积脏数据。
- **怎么实现**:一次模型调用批量裁决四种动作;两道保险——模型编造的目标 id 矫正为 ADD(防误删)、
  裁决失败降级为"全部新增 + 高相似去重"。

### 14. 多用户隔离
- **是什么**:每个用户(user_id)的记忆互相看不见。
- **优点**:防"帮我查另一个用户的地址"这类越权;客服场景一人一档。
- **怎么实现**:隔离**在存储层强制执行**(检索/删除都按 user_id 过滤,不靠上层自觉);user_id 由
  框架注入、**不进 prompt、模型看不见**(借鉴 Swarm 的 context_variables),防话术诱导。

## 五、大脑的大脑:多 Agent 编排

### 15. 三级分流(Router 分诊)
- **是什么**:一条诉求进来先分诊:寒暄/告知类走 **direct**(不调工具直接回复);单个领域的问题
  **直派专员**;跨领域复杂问题升级 **Supervisor** 中心调度。
- **优点**:**便宜的路先走**——不是所有输入都值得开全套编排。direct 1 次调用、专员直派约 4 次、
  中心调度约 11~15 次,成本递增。
- **怎么实现**:`Router` 一次模型调用输出目标;解析失败/拿不准就降级 supervisor(宁走贵路径不派错人)。

### 16. Plan-and-Execute + 动态重规划(Planner)
- **是什么**:Plan-and-Execute(先规划再执行):复杂诉求先拆成有序步骤,每步指派一个专员;某步
  失败就**动态重规划**(replan,重排剩余步骤)。
- **优点**:复杂任务可控、可追踪;失败能换路子(退款超时效走不通 → 改建工单转人工)。
- **怎么实现**:`Planner` 一次调用产出**线性步骤列表**(不做 DAG,客服步骤强依赖,线性够用且重规划
  语义干净);解析失败降级为单步计划。重规划全局限 1 次(刹车)。

### 17. Router + Supervisor 两种多 Agent 模式
- **是什么**:两种协作架构。Router = 入口分诊直派;Supervisor = 中心调度,按计划把步骤派给不同专员。
- **优点**:覆盖"简单直派"和"复杂协作"两种场景;三个专员(订单/售后/导购)**定义一次,两种模式
  复用**。
- **怎么实现**:关键取舍——**Supervisor 按 Planner 的计划派工,不让模型每步自由选人**(与 LangGraph
  的自由选人不同)。这样行为可预测、执行序列能被单测精确断言。专员之间不直接通话,一切经中枢
  (星型拓扑)。

### 18. Critic 自我纠错(Reflection)
- **是什么**:终稿质检员。方案产出后由一个独立的模型调用**对照证据审查**(诉求都回应了吗?有没有
  编造?),不合格带意见回炉重写一次。Reflection = 执行后自我反思。
- **优点**:提高输出质量,catch"漏了一项诉求""编造信息"这类错误;和"生成"分开,**评判比生成容易**。
- **怎么实现**:`Critic` 四项检查单;不合格只重写汇总(不重跑工具,便宜且幂等)≤1 次;二审仍不过
  就放行 + 留痕。解析失败视为通过(增益件不能变成新故障点)。

### 19. 专员通信协议 + 共享黑板(ScratchPad)
- **是什么**:专员之间靠"任务/回报"消息通信,失败时答复以"无法完成:"开头(相当于给自然语言加了
  个**状态码**)。步骤间的中间结果写在一块共享黑板 ScratchPad 上传递。
- **优点**:**显式传递优于隐式共享**——黑板上写了什么 trace 全看得见,可调试;订单专员查到的单号,
  售后专员下一步能直接用。
- **怎么实现**:编排器据"无法完成:"前缀判失败并触发重规划;ScratchPad 每步追加结论,超预算保新丢旧。

## 六、上生产:让它敢上线

### 20. 重试 + 指数退避(ReliableLLM)
- **是什么**:调用失败自动重试。指数退避(exponential backoff)= 等待时间翻倍增长(1 秒 → 2 秒 →
  4 秒),给对方喘息;加随机抖动(jitter)避免大家同时重试再次挤爆。
- **优点**:网络抖动、限流这类**临时故障能自愈**;退避避免"给快淹死的人继续泼水"。
- **怎么实现**:`ReliableLLM` 装饰器包住任意 `LLM`;**只重试该重试的错**(超时/限流/5xx),参数非法
  这类重试一万次也没用的直接上抛;`max_retries=3`(刹车)。**只装在 LLM 层,工具层不重试**(工具
  可能有副作用,见下条)。

### 21. 幂等(request_id)
- **是什么**:幂等(idempotent)= 同一操作执行一次和多次,效果相同。写操作工具带一个请求 id,服务端
  发现这个 id 处理过就直接返回上次结果,不重复执行。
- **优点**:让"审批通过后执行挂起的退款"这类可能被重放的动作**可以安心重试**,不会重复退款。
- **怎么实现**:`BaseTool.invoke(request_id=...)` 缓存首次成功结果(失败不缓存);id 由框架生成,
  不进模型能看到的 Schema(模型会忘、会编)。

### 22. 四层超时
- **是什么**:每个可能卡住的地方都设时间上限:①工具超时 ②LLM 单次调用超时 ③循环步数上限
  ④整任务总超时。
- **优点**:没有超时的调用像没闹钟的午睡;Agent 是多步循环,单步卡死会让整个任务无限悬挂。
- **怎么实现**:前三层各阶段已有,第④层整任务 deadline 在执行器的步骤边界检查——**超时不是失败,
  是转人工的信号**(任务终止入人工队列)。

### 23. 可观测:Trace + 结构化日志 + 指标
- **是什么**:Trace(轨迹追踪)= 一次任务从进到出每一步的完整记录(分诊/计划/派工/工具/耗时/
  token)。指标 = 从多次 trace 聚合出的成功率/平均步数/成本。
- **优点**:Agent 是多步自主决策,出错时"答案不对"要能定位到"第几步错了";**Trace 几乎是免费的**
  ——这些信息本来就流经代码,记下来即可。
- **怎么实现**:`Tracer` 结构化事件 + JSONL 落盘 + 广播(**一份事件流三个消费者**:存档/终端查看器/
  SSE 实时推送)。核心只加一个 `on_event` 只读钩子(缺省不影响任何行为)。结构化(JSON)而非自由
  文本,才能查询聚合。

### 24. 安全:六层纵深防御
- **是什么**:防 Prompt 注入(攻击者在输入或工具返回内容里塞"忽略之前的指令…"来篡改 Agent 行为)。
  没有银弹,只有层层设防:①输入检测 ②prompt 加固条款 ③工具返回边界标记(告诉模型"这是数据不是
  指令")④最小权限闸门 ⑤用户隔离 ⑥输出脱敏。
- **优点**:**赌注压在最小权限(第④层)而非检测**——注入终会有漏网之鱼,但高权限操作被审批锁死,
  被骗也难造成实质破坏。
- **怎么实现**:`BoundaryRegistry` 给工具返回包上边界标记(防"间接注入"——恶意指令藏在工具读到的
  内容里);`output_filter` 正则脱敏密钥/手机号;限流 + token 预算防刷量与失控。

### 25. HITL 人工介入(审批闸门 + 人工队列)★
- **是什么**:HITL(Human-in-the-loop,人在回路)。高权限动作不直接执行,被闸门**拦截 → 挂起入队 →
  等人工审批 → 放行后真正执行**。Agent 彻底办不了的诉求也升级同一个队列。这是业务场景的招牌能力。
- **优点**:**分级执行**落地——低权限自动办,高权限转人工;闭环(不是单向建工单,审批通过后系统
  真的把动作执行了)。
- **怎么实现**:`ApprovalGate` 是包在工具库外面的**装饰器**(拦截点在 Registry 层,专员/编排零改动);
  审批规则**配置驱动**(哪级要审批改配置,不写死 if/else);`HandoffQueue` 一个队列两个入口(审批 +
  兜底升级),放行时用 request_id 幂等执行,重复审批被拦。

### 26. 评估:LLM-as-Judge
- **是什么**:用另一个模型当裁判给 Agent 的自然语言输出打分(准确/完整/效率/安全四维)。因为
  "报告好不好"没法用 `==` 判断,人工逐个看又太慢。
- **优点**:把"我感觉挺好用"变成"25 例 76% 通过、四维可量化、改动可回归";评判比生成容易,和
  Critic 是同一个东西(在线返工 vs 离线评估,一次学习两处用)。
- **怎么实现**:`Judge` 一次调用四维打分;**防偏三件套**(裁判有偏见:偏爱靠前答案→交换顺序评两次;
  偏爱长答案→标准里写明"简洁是优点";偏爱同源模型→报告如实声明)。裁判**看轨迹不只看答案**
  (效率/安全维度必须看过程)。

### 27. 服务化:FastAPI + SSE
- **是什么**:把框架包成 HTTP 服务。FastAPI(异步 Web 框架,自动做请求校验和生成文档)。SSE
  (Server-Sent Events,服务器单向持续推送),用来把"正在搜索…正在分析…"这些中间步骤实时吐给用户。
- **优点**:外部任何系统 POST 一个请求就能用;长任务(几十秒)把中间步骤流式推送,体验天差地别;
  基于类型标注**自动生成交互式 API 文档**(`/docs`)。
- **怎么实现**:**同步核心 + 异步外壳**——Agent 同步循环跑在线程池,`on_event` 事件经队列桥接到 SSE
  生成器,核心不用改成 async。

### 28. 装饰器模式(插件不改造)+ 容器化
- **是什么**:装饰器(decorator)= 在不改原对象的前提下,包一层加功能。`ApprovalGate`/
  `BoundaryRegistry` 都是包在工具库外面的装饰器,实现同一个 `ToolRegistryLike` 接口。Docker
  容器化 = 把代码+依赖+环境打包成镜像,任何机器一致运行。
- **优点**:**加安全 = 加包装层,核心零改动**;装饰器可自由组合(边界标记∘审批闸门);Docker 解决
  "在我电脑上明明能跑"。
- **怎么实现**:装饰器透传 Registry 接口,专员完全不知道自己拿到的库被包过几层;十几行 Dockerfile,
  `.env` 外部注入不进镜像、`data/` 挂卷持久化。

## 收尾:一图看懂"这些技术怎么叠成一个系统"

```
LLM 接口(可换厂商)──► ReAct/FunctionCalling 循环 ──► 工具系统(BaseTool+权限)
        │                        │                          │
        │                   记忆(短期滑窗+摘要+三因子长期)      │
        │                        │                          │
        └──► 多 Agent 编排(Router 分诊 + Planner 计划 + Supervisor 调度 + Critic 质检)
                                 │
                    生产外壳:可靠性(重试/降级/幂等/超时)+ 可观测(Trace/指标)
                              + 安全(六层)+ HITL(审批闭环)+ 评估(Judge)+ 服务(FastAPI/Docker)
```

**每个技术不是孤立的**:LLM 接口让阶段六的降级成为可能,ReAct 循环成为专员内核,工具的权限标记
成为审批闸门的依据,记忆的用户隔离成为越权防御,多 Agent 编排成为控制系统——最后由 `AgentService`
门面组装成一个"换业务换实现而不动核心"的完整框架。三条信条(接口解耦 / 全链路降级 / 循环有刹车)
是把它们串起来的那根线。
