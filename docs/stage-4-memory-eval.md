# 阶段四评测报告 · 不同记忆策略的检索准确率对比

> 大纲阶段四产出⑥。评测脚本:`agent_framework/evaluation/memory_retrieval_eval.py`;
> 标注集:`agent_framework/evaluation/datasets/memory_retrieval.json`。
> 复跑:`python -m agent_framework.evaluation.memory_retrieval_eval`(约 34 条文本的
> 一次 embedding 批量调用,成本可忽略;检索过程零 LLM 调用)。

## 1. 评测设置

- **标注集**:手工构造 18 条预置记忆(u1 12 条 + 干扰用户 u2 6 条)+ 16 条查询,
  每条查询标注应命中的记忆 id。特意埋了一个**时近性陷阱**:新地址(上海,2 天前)与
  旧地址(北京,90 天前)语义几乎等价,只有时间能分辨。
- **指标**:Hit@1(top1 命中率)、Recall@3(top3 召回率),16 条查询平均。
- **控制变量**:embedding 统一 `text-embedding-3-small`;top_k=3;每条查询用**新鲜
  store**(检索会 touch 命中记录刷新时近性,复用 store 会互相污染);
  时间统一冻结在数据集的 `now`。
- **隔离红线**:评测同时断言所有命中必须属于查询用户(u2 的干扰记忆从未泄漏,通过)。

## 2. 结果(2026-07-07 实测)

| 策略(w_rel / w_rec / w_imp) | Hit@1 | Recall@3 |
|---|---|---|
| 无长期记忆(基线) | 0% | 0% |
| 纯相关性(1.0 / 0 / 0) | 81% | **100%** |
| **三因子·轻量(1.0 / 0.25 / 0.25)← 采纳为默认** | **94%** | **100%** |
| 三因子·初定(1.0 / 0.5 / 0.5) | 81% | 94% |
| 三因子·论文平权(1.0 / 1.0 / 1.0) | 38% | 75% |

## 3. 失败模式分析(比数字更重要)

- **纯相关性输在哪**:新旧地址陷阱全军覆没——「寄到我常用的地址」两条查询都返回了
  90 天前的北京旧地址(m02)。语义上新旧两条几乎无法区分,这正是时近性因子存在的理由。
- **初定 0.5/0.5 输在哪**:时近/重要加成过强,「新且重要」的记忆会插队弱相关查询——
  问会员等级,被 3 天前的「补发优惠券承诺」(重要性 9)抢了 top1;问快递偏好,被
  2 天前的新地址(重要性 9)抢走。加分项盖过了主排序。
- **论文平权为什么最差(38%)**:三项平权后,一条又新又重要的记忆几乎霸榜所有查询
  (m01 新地址插队了 6 条无关查询)。Generative Agents 的场景是「智能体回忆自己的
  生活流水」,记忆之间相关性普遍偏低、时间跨度大,平权合理;客服场景查询意图明确,
  相关性必须主导。**结论:论文公式可以搬,权重不能照搬,要按场景实测。**
- **轻量 0.25/0.25 的平衡点**:加成刚好够在「语义打平」时用时间分胜负(赢下地址陷阱),
  又不足以让无关记忆插队(会员/快递/发票全部守住)。唯一失败的一条
  (「什么时间段送货比较方便」→ 命中快递偏好而非收货时间偏好)是纯 embedding
  语义区分度问题,四种策略全错,与权重无关。

## 4. 结论与动作

1. **默认权重改为 `1.0 / 0.25 / 0.25`**(`config.py` 已改,评审初定的 0.5/0.5 被
   数据推翻——这正是「先拍板默认值、评测说了算」流程的价值)。
2. 三因子确实优于纯余弦,但优势**只该来自平局裁决**:辅助因子权重宁小勿大。
3. Recall@3 上纯相关性也有 100%:top3 给到模型时排序错误还能被模型自己纠偏,
   但 top1 错会直接污染「已知信息」的第一印象——Hit@1 是更贴近实际伤害的指标。
4. 局限:16 条查询的小样本,结论是方向性的;权重是在本集上选的,存在过拟合风险,
   阶段六可在更大评测集上复核。
