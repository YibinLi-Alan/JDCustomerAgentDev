# 项目综述 · 从零构建京东客服 Agent 框架(High-Level Description)

> 一份高层次的项目描述:业务是什么、怎么实现的、用什么工具做的、依据了哪些论文、
> 完成了什么(带量化数据)、实现了哪些功能、还有什么局限。适合用于答辩、汇报或简历。

## 一句话概述

在一个 4–6 周的实习周期内,**不依赖 LangChain/AutoGen 等成熟框架,从零手写**一个模块化、
可复用的通用 Agent 框架,并落地为"京东客服 Agent";全程借助 **Claude Code(Anthropic
的智能体编码 CLI)** 以"虚拟多 Agent 工程团队"的方式推进,最终交付 **9 个子包、226 条
离线单测全绿、端到端评测 76% 通过、安全攻击集 9/9 防住** 的完整系统。

## 1. 业务是什么

**场景:京东客服 Agent**(依据业务案例文档 `frame/给你一个直观的例子.docx`)。
- **输入**:用户的一条反馈 / 问题(自然语言诉求);
- **输出**:不是单句回答,而是一套**解决方案(solution plan)**——要做哪些动作 + 支撑材料;
- **核心业务规则:分级执行 + 人在回路(HITL)**——低权限动作(查订单/物流/商品)Agent
  直接办并返回结果;高权限动作(退款/取消订单/建工单)**触发人工审批**;Agent 彻底办不了的
  诉求带完整轨迹**升级到人工队列**。

选这个场景是因为它**天然用上一个 Agent 框架的全部模块**:自主循环、工具调用、订单/用户记忆、
多步方案规划、问题分流协作、权限分级与审批——是检验"功能完整"的试金石。

## 2. 我是怎么实现的

**方法论:从零逐层构建,严格"设计文档先行"**。不套用现成框架,而是逐层手写以真正理解每个
部件为何这样设计。整个项目按固定的**六阶段课程**推进,每阶段"**先写设计文档 → 评审拍板 →
再编码 → 测试全绿 → 提交**":

| 阶段 | 主题 | 交付 |
|---|---|---|
| 一 | 基础认知 | LLM 接口抽象 + 可切换 provider + 配置 + Prompt 实验 |
| 二 | 最小 Agent 循环 | ReAct(推理-行动-观察)+ 步数守卫 + 错误自愈 |
| 三 | Tool Use 系统 | BaseTool + Registry + 原生 Function Calling + 11 工具 |
| 四 | Memory | 短期滑窗 + 递归摘要 + 长期三因子检索 + 多用户隔离 |
| 五 | Planning & Multi-Agent | Router + Supervisor + Planner + 三专员 + Critic |
| 六 | 生产化 | 可靠性 / 可观测 / 评估 / 安全 / HITL 闭环 / FastAPI + Docker |

**三条贯穿全程的工程信条**:①核心只认接口,加能力=加实现(会变的东西全藏在 Protocol/ABC 后);
②全链路降级永不炸(任何失败折叠成数据继续走);③凡是循环必有刹车(每个重复处都有次数上限)。

## 3. 用什么工具做的:Claude Code

全程由 **Claude Code(Anthropic 官方的智能体命令行编码工具)** 完成设计、编码、测试、评测与
文档。使用方式上有两个特点:

- **"虚拟大厂 Agent 团队"操作模型**:主循环扮演 Orchestrator / Tech Lead,按阶段派出专职
  子 Agent(Architect / Core Runtime / Tools / Memory / Orchestration / Platform / Safety /
  QA / DevRel),模块与负责人一一对应,保证"没有无主之地"。代码审查阶段也用两个子 Agent
  分头审查、交叉验证。
- **知识库沉淀**:用 Obsidian 知识库(`AgentKnowledge/`)全程记录关键决策、对比阅读、阶段
  总结、踩坑与任务日志——设计论文、架构图等文档都基于这些真实日志,而非事后补写。

配套工程工具:Python 3.11 · Pydantic(数据校验)· ChromaDB(向量库)· FastAPI + uvicorn
(服务化)· pytest / ruff / black / mypy(测试与静态检查)· Docker(容器化)。

## 4. 依据了哪些论文与理论

框架"每一块设计都有出处",做了系统的对比阅读(comparative reading):

| 技术点 | 依据 |
|---|---|
| 自主循环 | **ReAct**(Reasoning + Acting)· Lilian Weng《LLM Powered Autonomous Agents》 |
| 长期记忆三因子检索 | **Generative Agents**(斯坦福小镇,arXiv 2304.03442):相关性+时近性+重要性加权、检索"保鲜" |
| 上下文分区管理 | **MemGPT**(arXiv 2310.08560):把上下文当操作系统内存分区调度 |
| 记忆增删改写入 | **Mem0**:提炼事实 + ADD/UPDATE/DELETE 写入决策 |
| 先规划再执行 / 重规划 | **Plan-and-Solve** · LangGraph plan-execute |
| 分诊 / handoff | **OpenAI Swarm**:handoff、context_variables(user_id 不进 prompt) |
| 中心调度 | **LangGraph Supervisor**(并做了"按计划派工不让 LLM 自由选人"的关键改动) |
| 自我纠错 | **Reflexion**(arXiv 2303.11366):错误信号转语言反馈 |
| 业务 SOP 编码进分工 | **MetaGPT**(arXiv 2308.00352) |
| 任务规划四段式 | **HuggingGPT**(arXiv 2303.17580) |
| 群聊式多 Agent(对比不采用) | **AutoGen**(arXiv 2308.08155) |
| 离线评估 | **Judging LLM-as-a-Judge**(arXiv 2306.05685):位置偏见/啰嗦偏好等裁判偏差与对策 |
| 安全防护 | **OWASP Top 10 for LLM**:Prompt 注入排第一 |
| 生产化实践 | Chip Huyen《Building LLM Applications for Production》· Eugene Yan《LLM Patterns》 |
| 工具设计 | **LangChain** Tool / BaseTool 思想(补强其缺失的统一 Registry) |

**一个方法论亮点**:三因子检索的权重不照搬论文——论文的等权在客服场景实测只有 38% 命中率,
我们用自建评测集调到 **94%**,并写进设计文档说明"公式可以搬,权重必须按场景实测"。

## 6. 实现了什么样的功能

- **可换厂商的 LLM 内核**:Claude/OpenAI 双 provider,换厂商改一行配置;带指数退避重试 + 主备降级。
- **两种 Agent 循环**:原生 Function Calling(并行调用)+ 文本 ReAct(兜底),坏输出能自我纠正。
- **完整工具系统**:强类型参数校验(strict mode)、超时、权限分级、错误折叠、`@tool` 一行造工具。
- **完整记忆系统**:短期滑窗 + 递归摘要 + 长期语义检索 + 记忆增删改 + 多用户存储层隔离 + 跨会话持久化。
- **多 Agent 控制系统**:三级分流(直接回复/专员直派/中心调度)+ 计划驱动派工 + 动态重规划 +
  终稿质检回炉。
- **生产级外壳**:结构化 Trace + 指标仪表盘;六层注入纵深防御 + 输出脱敏 + 限流;LLM-as-Judge
  自动评测;FastAPI + SSE + Docker。
- **业务闭环(招牌能力)**:**分级执行 + HITL 人工审批**——低权限自动办、高权限拦截挂起等人工
  放行后真正执行、办不了的带轨迹转人工。

## 7. 局限性(如实记录,不假装无懈可击)

**这是一个学习性质的实习项目,准则是"功能完整 > 工业级打磨"**——每个模块都真实存在、能跑、
能演示,但有意识地在以下方面做了取舍,以下如实说明:

1. **业务数据是 mock**:订单/商品/物流是内存假数据(`JDMockStore`),未接京东真实 API;且该
   数据层目前是"鸭子式字典访问",尚未接口化——接生产数据库前需先补一个仓储接口(推荐
   PostgreSQL + pgvector,详见 README 存储设计)。
2. **评测样本小、单跑**:76% 基于 25 例单次运行,只作**改动前后回归对比**而非绝对结论;裁判与
   被评是同款模型(gpt-5.4-mini)存在**自我增强偏见**;评测跑在无长期记忆配置下,故记忆类
   用例失败。
3. **进程内、单实例**:限流、token 预算、人工审批队列(JSON 文件)都是进程内实现,重启清零、
   多实例不共享,生产需外置到 Redis / 数据库。
4. **安全是纵深而非银弹**:注入检测是启发式正则,变体话术会漏;输出脱敏只防原样复读密钥,防不了
   改写后泄漏;9/9 是特定 9 例攻击集的结果,不代表覆盖全部攻击面。真正的兜底是"最小权限闸门"。
5. **不做重量级能力**:无任务队列(分钟级长任务未支持,几十秒级靠 SSE 兜底);未接 Langfuse 等
   可观测平台(自建极简 Trace,可对接);模型用便宜的 gpt-5.4-mini 省成本,行为稳定性随模型
   强弱变化。
6. **教学取舍**:ReAct 兜底循环的 few-shot 写死了具体工具名;多 Agent 只实现 Router+Supervisor
   两种模式(未做层级/群聊模式,已在文档说明理由)。

> 能量化结果、也敢写清局限,本身是工程成熟度的体现——这比报一个漂亮的满分更有说服力。
