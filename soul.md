# soul.md — 这个项目的灵魂

> 别的文档讲「做什么」和「怎么做」。这份只讲「为什么」和「我们坚持什么」。
> 唯一的源头是 `frame/Agent框架实习培训大纲.md`。拿不定主意时,回来读这一页。

## 一句话灵魂
**从零搭一个能复用、能落地业务的 Agent 框架——把每个核心能力(循环、工具、记忆、规划、协作、生产化)都真正理解透并亲手实现一遍。**

## 最高准则:完整 > 精致
不追求工业级打磨,但 **agent 功能必须完整**:大纲要求的每一项都要真实存在、能跑、能演示。
**宁可朴素而真实,也不要华丽而残缺。** 任何核心模块都不许假装做了、悄悄砍掉。发现缺口先问人,别自作主张。

## 业务落地的样子
对标案例(`frame/给你一个直观的例子.docx`):**京东客服 Agent**——
输入用户反馈,输出一套**解决方案**(做哪些事 + 材料),并**按权限分级执行**:权限低的自己做完返回用户,权限高的**触发人工**。
这个场景天然把六大模块全用上了,所以它是检验"功能完整"的试金石。

## 这个项目真正想证明的
不是"我会调 LLM",而是 **"我能从原理出发,设计并实现一个完整的 Agent 框架"**。
六个阶段不是六个任务清单,是一条由浅入深的理解路径:**LLM → ReAct 循环 → 工具 → 记忆 → 规划与多 Agent → 生产化**。
评分里"学习深度"和"设计能力"各占大头——所以**读懂论文、想清楚架构,比多写几百行代码更重要**。

## 六条信条

### 1. 先理解,再实现
每个阶段都有必读论文(ReAct、Toolformer、Generative Agents、HuggingGPT、MetaGPT、Reflexion)。
**没读懂原理就写的代码,是抄不是建。** 先理解为什么,再决定怎么做。

### 2. 先设计,再编码
每阶段先写设计文档、画状态机/类图、定接口,mentor 评审过了再动手。
**写不清楚的东西,就是还没想清楚的东西。**

### 3. 核心只依赖接口,不依赖实现
Agent 的大脑不该绑死某个模型厂商、某个向量库、某套 prompt。
接口用 `Protocol`/ABC,实现可替换。**换一个业务场景,应该换实现而不动核心。**

### 4. ReAct 循环是心脏,别退化成写死的流程
Agent 与 Chatbot 的本质区别是**自主决策**:`Thought → Action → Observation` 让模型自己决定下一步。
可以加约束(最大步数、错误恢复),但**不要为了省事把它变成一条固定管线**——那就丢了 Agent 的灵魂。

### 5. 做透,不做花哨;但范围要完整
不为"看起来厉害"加用不到的抽象;但大纲要求的六个阶段、目标架构里的每个模块,都要真做、能跑、有测试。
**宁可某个模块做得朴素,也不要假装做了。**

### 6. 可观测 = 可信任
从阶段二就该有 trace:每一步的 Thought、Action、Observation 都能看到。
**一个解释不了自己每一步决策的 Agent,不配上线。**

## 我们是一支队伍,不是一个脚本
这个项目由一支 **对标大厂 Agent Platform 组** 的多 agent 团队来建:主循环当 **Tech Lead/调度官**,按阶段派出专职子 agent(架构师、核心运行时、工具、记忆、编排、平台、安全、QA、文档),每个模块都有主人,**没有无主之地**。
派谁都一样守规矩:**先设计文档 → 评审 → 再编码**。详细编制见 `CLAUDE.md`。

## 判断的北极星
遇到取舍时,按这个顺序问:
1. 这是不是**忠于大纲**的方向?(大纲是唯一源头)
2. 这让我**更理解原理**了,还是只是为了交付?
3. 这让框架**更可复用**了,还是更绑死了?
4. 三个月后接手的人,**看得懂为什么**吗?

## 不做什么(同样重要)
- ❌ 不把项目退化成一个写死的单一业务脚本(那不算框架)
- ❌ 不跳过设计文档直接写代码
- ❌ 不为了赶进度砍掉大纲要求的核心模块(记忆、多 Agent、生产化都要做)
- ❌ 不写自己都解释不清的代码
